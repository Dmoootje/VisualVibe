**Building an app** usually begins with an idea: a process could be faster, clients need a dedicated portal, or an existing mix of spreadsheets and emails has become unmanageable. The biggest mistake is to turn that idea immediately into a long list of screens and features.

A useful app starts by defining the problem, the people who will use it and the first version that delivers enough value to test. Only then should you choose the technology, platform and development sequence.

This guide helps you prepare an app project realistically, without overlooking important technical or organisational questions.

## Start with the problem, not the feature

"We need an app" is not yet a project objective. First describe what currently goes wrong.

Examples include:

- employees entering the same information in three different places;
- clients repeatedly calling for status updates;
- quotations being compiled and followed up manually;
- planning being scattered across calendars, messages and spreadsheets;
- documents being difficult to retrieve;
- an existing platform not fitting the way the organisation works.

A useful problem statement makes the result measurable. For example: "An employee must be able to process a request in under five minutes without re-entering any information."

## Who are the real users?

An app may serve clients, employees, administrators, partners or suppliers. Each group has different tasks, permissions and levels of technical experience.

For each user, describe:

1. the task they perform;
2. the information they need;
3. the decisions they make;
4. the mistakes that commonly occur;
5. the device and environment in which they use the app.

A warehouse employee wearing gloves and using a smartphone has different needs from a business owner analysing reports on a desktop computer.

## Choose the smallest valuable first version

A Minimum Viable Product, or MVP, is not simply a cheap or unfinished app. It is the smallest version with which you can test the core value safely and credibly.

A sound MVP includes:

- one clear primary task;
- the essential user roles;
- adequate security and logging;
- an admin workflow for errors and exceptions;
- measurements for evaluating use and results.

Extensive reports, dozens of settings and rare exceptions can often follow later. Every additional feature increases the design, development and testing effort.

## Web app, mobile app or both?

A web application runs in a browser and is usually quick to make available on desktop, tablet and smartphone. A mobile app is installed through an app store or managed distribution channel and can use device hardware more deeply.

A web app is often suitable when:

- users mainly manage forms, dashboards and records;
- one central version is important;
- desktop use plays a major role;
- rapid updates without app-store review are desirable.

A mobile app is more likely to fit when:

- offline use is important;
- push notifications are a core feature;
- the camera, location, Bluetooth or other device hardware is used intensively;
- users open the app very frequently while on the move.

Read our detailed comparison of a [web app or mobile app](/en/knowledge-base/web-design/web-app-or-mobile-app/) for more help with this decision.

## UX and app design before development

A clickable prototype makes the flow visible before the back end, database and integrations are built. This matters because many expensive mistakes are not programming errors, but incorrect assumptions about users.

A UX/UI process can include:

- user flows;
- wireframes;
- a clickable prototype;
- a visual design system;
- loading, error and empty states;
- responsive behaviour across different screens.

Explore our [app design and UX/UI service](/en/services/custom-software/app-design-ux-ui/) if you want to validate the idea visually and functionally first.

## The technical foundations of an app

A modern app often consists of several layers:

- **front end:** screens and interactions;
- **back end:** business logic and secure processing;
- **database:** structured data;
- **authentication:** login, roles and permissions;
- **APIs:** communication with other systems;
- **administration environment:** management of users, content and exceptions;
- **logging and monitoring:** tracking errors and important actions.

The choice of technology follows requirements around scale, team expertise, offline use, data, hosting and integrations. Do not choose a framework simply because it is popular without assessing whether the project genuinely benefits from it.

## Roles, permissions and privacy

Decide early which data each user may view or change. Merely hiding buttons in the interface is not security. Permissions must also be enforced in the back end.

Consider matters such as:

- who creates and deletes accounts;
- which information clients may see about one another;
- how administrators can audit actions;
- which data is retained;
- how exports and deletion requests work;
- how sensitive information is encrypted;
- how test data is separated from production data.

Adding privacy and security afterwards costs more than including them in the data model from the outset.

## Integrations with existing software

An app rarely operates entirely on its own. Possible integrations include CRM, accounting, planning, email, payments, documents, analytics and external data sources.

For every integration, establish:

- which system is the source of truth;
- which data moves in which direction;
- how duplicate records are prevented;
- what happens after an error or timeout;
- how changes are logged;
- who manages access keys.

Sometimes a focused [API integration and automation](/en/services/custom-software/api-integrations-and-automation/) is enough, so you do not need to build an entirely new app.

## What determines the cost of an app?

Cost is not determined by the number of screens alone. Important factors include:

- the number of user roles;
- the complexity of business rules;
- the volume and sensitivity of data;
- integrations with external systems;
- offline operation and synchronisation;
- the administration environment and reporting;
- migration of existing data;
- testing, security and documentation;
- the support required after launch.

Our guide to [how much it costs to build an app](/en/knowledge-base/web-design/how-much-does-an-app-cost/) provides a practical budget breakdown.

## Test real scenarios

An app can work technically and still be unusable. Test complete real-life situations, not just individual buttons.

Examples include:

- a new client creates an account and forgets the password;
- an employee enters incomplete information;
- a payment succeeds but its confirmation is delayed;
- two users edit the same record;
- an external integration is temporarily unavailable;
- an administrator must correct an error manually.

Good tests cover normal flows, exceptions and permission checks.

## Launch begins the usage phase

Assumptions become visible after launch. Users may take different routes from those expected or request features that seemed unimportant during the briefing.

Plan for:

- monitoring and error alerts;
- a support process;
- small improvements after the initial usage period;
- dependency and security maintenance;
- priorities for phase two;
- evaluation of use and outcomes.

An app without maintenance ages just as a website does. Operating systems, browsers, APIs and security requirements change.

## AI in an app: only for a bounded task

AI can summarise documents, answer questions, classify data or prepare suggestions. It is not a substitute for clear processes and reliable data.

Decide in advance:

- which task AI performs;
- which source information it uses;
- how quality is tested;
- when human review is required;
- what happens when the model is uncertain;
- which data is sent to external models.

Read [building an AI application](/en/knowledge-base/web-design/building-an-ai-application/) for a practical approach.

## Summary

Building an app is a product and process project, not merely a programming assignment. Start with a specific problem, design around real users, limit the first version and account for permissions, data, integrations and maintenance from the beginning.
