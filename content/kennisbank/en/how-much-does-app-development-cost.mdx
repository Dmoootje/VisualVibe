**How much does it cost to have an app made?** There is no reliable standard amount for this without knowing the users, functions, data and integrations. Two apps with ten screens can be completely different technically. A simple information flow is different from a platform with payments, roles, documents and synchronisation with external systems.

A good price estimate therefore starts with scope and risk. In this article you will learn which components determine the budget and how to prevent a cheap quote from turning into an expensive unfinished project later.

## Why a price per screen means little

A screen can be a simple text page, but also a dashboard with filters, permissions, graphs and real-time data. The visible design is only one layer. Behind an app there are often database rules, authentication, validation, logging and integrations.

Therefore, do not just compare quotes on numbers of screens. Ask what user scenarios, back-end functions, test cases, and delivery components are included.

## The most important price factors

### 1. User roles and access rights

An app for one admin is simpler than a platform for customers, employees, partners and super admins. Each role requires rules about what someone can see, add, change and delete.

Complexity mainly arises when rights differ per organization, file or record. Those rules need to be enforced and tested in the back-end.

### 2. Data model and business logic

The database must support real operation. Think of customers, projects, tasks, statuses, documents, products, planning and invoices.

Price increases when:

- data have many mutual relationships;
- statuses may only change under conditions;
- historical versions must be preserved;
- calculations and exceptions are important;
- multiple users change data at the same time.

A good data model prevents expensive conversions later.

### 3. Links with external systems

Links with CRM, accounting, payments, calendar, e-mail, maps or other software require analysis and error handling. An API can be well documented, but also limited, paid or unreliable.

For every integration, it must be clear what happens when the external system temporarily does not respond. Logging, retries, and manual recovery options take work, but prevent silent data losses.

### 4. Web app or mobile app

A responsive web app can serve many devices with one code base. A native mobile app may require additional development for iOS, Android, app stores, device permissions and versions.

Do not automatically commission two mobile apps when a web application performs the job well. Read [web application or mobile app](/en/kennisbank/webdesign/web-app-or-mobile-app/) for the trade-off.

### 5. Offline use and synchronization

Working offline is technically much more complex than just showing data locally. The app should save changes, sync later, and resolve conflicts when multiple users modify the same item.

Include offline functionality only when the work environment really requires it.

### 6. UX/UI and prototype

Wireframes and prototypes take time, but save development when they make incorrect flows visible early. A good design also includes loading states, error messages, blank screens, mobile variants and accessible behavior.

A prototype is especially valuable in complex workflows or when multiple stakeholders have different expectations.

### 7. Management environment

Almost every app needs management. Someone needs to block users, correct data, modify content, create exports or fix errors.

A management environment is not an automatic free by-product. Determine which actions administrators need and which actions need to be logged.

### 8. Security and privacy

Sensitive data, payments, medical information or complex roles increase analysis and testing work. Security includes authentication, authorization, encryption, logging, backups and incident response.

A quote that only mentions security as one general point is difficult to assess. Ask what concrete measures and tests are provided.

### 9. Data migration

Existing spreadsheets, CRM exports or old databases are rarely perfectly cleaned up. Import requires mapping, validation, test migrations, and a plan for duplicate or missing data.

The amount of records is not the only factor. Data quality determines much of the work.

### 10. Maintenance and further development

After launch, hosting, monitoring, updates, support and improvements remain necessary. External APIs and dependencies change. So don't reserve the entire budget for the first go-live.

## Work with phases instead of one gigantic scope

A feasible project can consist of four budget layers.

### Phase 1: analysis and prototype

Goal: make risks and assumptions visible.

Possible output:

- functional analysis;
- user roles;
- core flow;
- data model in outline;
- technical advice;
- clickable prototype;
- MVP scope and estimate.

### Phase 2: MVP

Goal: test the core value in production.

The MVP contains only necessary functions, but sufficient security, logging and management to allow real users.

### Phase 3: integrations and deepening

Proof of use will be followed by additional links, reports, automation, and exceptions.

### Phase 4: Optimization and Scale

Here you improve performance, onboarding, analytics and processes based on real data.

## How do you compare app quotes?

Check whether each quote answers these questions:

- Which user roles are included?
- Which concrete flows are built?
- Is UX/UI included?
- Who builds and manages the back-end?
- What links are included?
- How are errors and exceptions handled?
- What tests are performed?
- Who owns code and accounts?
- Which hosting and usage costs are external?
- What documentation is provided?
- What is covered by warranty and what is covered by maintenance?
- How are scope changes priced?

A low price can be the result of an efficient solution, but also of missing parts. Therefore, compare the demarcation, not just the final amount.

## Common budget mistakes

### Want everything in version one

A large initial scope increases lead time and makes feedback expensive. First build the functions that enable the main task.

### Forgot external costs

Think of hosting, email, storage, maps, payment providers, AI usage, app store accounts and premium APIs. Some costs grow with usage.

### No budget for content and data

Texts, product data, imports, document structure and test data must be prepared. Bad source data slows down development.

### No owner within the company

A developer can guide technical choices, but someone within the organization must determine priorities and explain processes. Lack of decisions causes delays and extra work.

## Can you get a fixed price?

A fixed price is possible for a well-defined phase. For a new product with a lot of uncertainty, a short paid analysis is often fairer than an apparently fixed total price with broad assumptions.

A combination works well:

1. fixed price for analysis and prototype;
2. fixed or limited bandwidth for a clearly described MVP;
3. loose priorities for later iterations.

## Summary

The costs of an app are determined by users, data, business rules, connections, platform, management, security and maintenance. A realistic budget is created by describing the core flow and dividing the project into phases.
